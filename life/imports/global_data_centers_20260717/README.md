# Global data-center public location baseline

This import is the reproducible location layer for Brian's global AI data-center
research. It queries every current OpenStreetMap object matching any of these tags:

- `telecom=data_center`
- `building=data_center`
- `industrial=data_centre`
- `industrial=data_center`

Run from the repository root:

```sh
python3 scripts/build_global_data_center_registry.py
```

The registry deliberately leaves GPU count, IT/utility power capacity, and power
equipment empty. Those fields are populated only in the separate deep-research
layer when a company filing, operator page, utility filing, permit, or other
traceable source supports the claim.

## Coverage boundary

"Every" in this file means every object returned by the documented OSM query at
the retrieval time. It does **not** mean every physical data center worldwide.
Private enterprise rooms, undisclosed hyperscale facilities, military sites, and
poorly mapped regions are systematically undercounted. A campus can also appear as
several building objects.

PeeringDB is used only as a non-republished cross-check because its acceptable-use
policy restricts bulk redistribution. On 2026-07-17 its public site reported 5,858
facilities, versus the independently queryable OSM tag baseline in this directory.

## Official operator registries

Operator-page registries in this directory are separate from the OSM-derived
database and retain their own source and disclosure boundaries:

- `cyrusone_official_location_pages.jsonl` and its summary are generated by
  `scripts/build_cyrusone_official_registry.py`.
- `qts_official_location_pages.jsonl` and its summary are generated by
  `scripts/build_qts_official_registry.py`.
- `microsoft_official_region_registry.jsonl` and its summary are generated by
  `scripts/build_microsoft_official_region_registry.py`. This registry unions
  Microsoft's Datacenters Globe records with the Azure Learn public-cloud table
  and explicitly preserves source-only rows and upstream anomalies.
- `aws_official_region_registry.jsonl` and its summary are generated by
  `scripts/build_aws_official_region_registry.py`. This registry reconciles the
  39-Region/123-AZ global marketing scope with the standard-account 34-Region/
  108-AZ-ID documentation scope and the GovCloud, China, and European Sovereign
  partitions. Marketing map IDs remain distinct from programmatic Region codes.
- `google_official_location_registry.jsonl` and its summary are generated by
  `scripts/build_google_official_location_registry.py`. This registry reconciles
  Google's hidden map payload, visible list, FAQ roster, and selected detail
  pages. It preserves the Canelones lifecycle conflict and shows why the static
  30-location/11-country counters likely predate the 2025 Waltham Cross opening.
- `oracle_official_region_registry.jsonl` and its summary are generated by
  `scripts/build_oracle_official_region_registry.py`. This registry reconciles
  Oracle's commercial, sovereign, and government marketing rows with the
  commercial OCI technical Region table. It preserves Hyderabad/Batam and Madrid
  code conflicts, separates 27 live commercial countries from the 28-country
  live-plus-planned headline scope, and treats 55 Availability Domains only as a
  physical-building lower bound against Oracle's separate 147-active claim.

These registries may mix operating, development, and future facilities. Published
capacity, area, investment, and equipment statements are not treated as current
energized, leased, utilized, or billed capacity unless a separate source supports
that lifecycle conversion. Microsoft Azure regions, availability zones, and
provider map coordinates are also not one-to-one physical buildings, exact street
addresses, or ownership records. Google location labels and detail pages can also
cover multiple physical sites, and Google map coordinates are retained only as
provider map anchors.
