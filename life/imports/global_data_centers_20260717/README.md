# Global data-center public location baseline

This import is the reproducible location layer for Brian's global AI data-center
research. It queries every current OpenStreetMap object matching any of these tags:

- `telecom=data_center`
- `building=data_center`
- `industrial=data_centre`
- `industrial=data_center`

Run from the repository root:

```sh
python3 scripts/build_global_data_center_registry.py
```

The registry deliberately leaves GPU count, IT/utility power capacity, and power
equipment empty. Those fields are populated only in the separate deep-research
layer when a company filing, operator page, utility filing, permit, or other
traceable source supports the claim.

## Coverage boundary

"Every" in this file means every object returned by the documented OSM query at
the retrieval time. It does **not** mean every physical data center worldwide.
Private enterprise rooms, undisclosed hyperscale facilities, military sites, and
poorly mapped regions are systematically undercounted. A campus can also appear as
several building objects.

PeeringDB is used only as a non-republished cross-check because its acceptable-use
policy restricts bulk redistribution. On 2026-07-17 its public site reported 5,858
facilities, versus the independently queryable OSM tag baseline in this directory.

These registries may mix operating, development, and future facilities. Published
capacity, area, investment, and equipment statements are not treated as current
energized, leased, utilized, or billed capacity unless a separate source supports
that lifecycle conversion. Microsoft Azure regions, availability zones, and
provider map coordinates are also not one-to-one physical buildings, exact street
addresses, or ownership records. Google location labels and detail pages can also
cover multiple physical sites, and Google map coordinates are retained only as
provider map anchors. Apple's eight owned labels are likewise sites or campuses,
not eight proven physical buildings; Apple explicitly describes multiple buildings
at Maiden, aggregates both Chinese sites in one energy row, and does not publish
the number of colocation or third-party PCC facilities. Alibaba Cloud defines a
zone as a physical area made of one or multiple scattered data centers, so its 130
published zone records are neither a building count nor a capacity total. Tencent
calls an AZ a physical data center, but its service table still omits building and
campus boundaries, addresses, ownership, capacities, lifecycle, and utilization.
Volcano Engine's Region/AZ counts vary by product, and its API-endpoint guide
explicitly says an endpoint is not necessarily the product's direct service
location; the nine-ID source union must therefore not be read as nine physical
Regions, campuses, or buildings. Neocloud service Regions, partner facilities,
GPU product ranges, financing announcements, and host leases have the same
scope problem: use the per-record lifecycle and ownership fields before comparing
  capacity or economics. The physical-operator crosswalk likewise does not rank
facility counts or sum capacity because buildings, facility IDs, campuses,
markets, design envelopes, live IT load, and utilized load are different objects.
The supply-chain crosswalk also requires one accounting basis, value pool,
commercial lifecycle, and site or segment allocation before comparing economics.
The accelerator ledger additionally requires one hardware unit, ownership state,
date, physicality class, lifecycle, site scope, and resolved overlap group.
