Pain 1

검증 loop에서 요구의 빈틈이 먼저 드러날 수 있다

failing log mismatch missing assumption unclear acceptance owner decision
Agent가 design intent를 추측해 고치는 것이 아니라, 어떤 요구가 불명확한지와 어떤 evidence가 있는지를 분리한다.
Pain 2

AI workflow에서는 생성 속도와 운영 복잡도가 함께 늘어날 수 있다

속도가 만든 것
  • ReAct loop
  • RAG / memory
  • subagent
  • Plan mode
새 failure mode
  • hallucinated observation
  • infinite loop
  • stale state
  • context pollution

검증 triage 속도를 높이더라도 loop control, context, evidence, stale state 문제를 함께 봐야 한다.

Pain 3

마음대로 만들게 해도, 전부 묻게 해도 문제가 생긴다

1차: 의도만 전달 -> Agent가 빈칸을 마음대로 채움
2차: 모호하면 질문 -> 질문 폭주
3차: test pass 확인 -> 의도 충족 장치가 필요해 보임
4차: 같은 실수 반복 -> rule / skill / wiki / ontology가 도움
검증 agent는 evidence로 판단 가능한 것, 사람이 결정할 것, intent-conformance 확인 지점을 나누는 쪽이 좋아 보인다.

Practical Loop

검증 evidence를 구조화한다

failing command / log
-> first evidence line
-> suspected area
-> related spec/test/RTL context
-> safe next check
-> human decision needed

Agent가 RTL을 고치는 장면보다, 검증 실패를 구조화하고 사람이 볼 결정을 표시하는 장면이 더 현실에 맞을 수 있다.
